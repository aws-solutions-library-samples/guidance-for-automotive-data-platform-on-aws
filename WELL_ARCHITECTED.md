# Well-Architected Framework: Automotive Data Platform on AWS

This document describes how the Automotive Data Platform aligns with the AWS Well-Architected Framework across all guidance projects.

## Operational Excellence

### What service(s) are you using to enhance operational excellence?

This platform uses AWS CloudFormation (via CDK), Amazon CloudWatch, AWS X-Ray, AWS Glue Data Catalog, Amazon Athena query history, and AWS Step Functions for operational excellence. CloudFormation provides infrastructure as code for consistent deployments across all guidance projects. CloudWatch captures logs and metrics from Lambda functions, Glue jobs, SageMaker training/inference, and Bedrock agents. X-Ray traces distributed requests through the predictive maintenance API and ML inference pipeline. The Glue Data Catalog maintains metadata versioning and schema evolution tracking across both Customer 360 and Predictive Maintenance datasets. Athena query history provides audit trails of all data access patterns. Step Functions visualizes workflows for ML training pipelines, batch inference jobs, and Bedrock agent interactions.

### How do these services help the user with operational excellence?

These services enable comprehensive observability and automated operations management across the entire platform. CloudWatch Logs aggregate all application logs in a centralized location, making it easy to search across Customer 360 analytics queries, predictive maintenance inference requests, and Bedrock agent conversations. X-Ray distributed tracing shows the complete request path for tire predictions, from API Gateway through Lambda preprocessing, SageMaker inference, and alert generation, identifying performance bottlenecks at each step. The Glue Data Catalog automatically tracks schema changes when crawlers run, preventing data pipeline failures from schema drift in both customer analytics and telemetry data. Athena query history provides a complete audit trail supporting compliance requirements and troubleshooting of slow queries. Step Functions visualizes complex workflows including ML training (data validation, feature engineering, model training, evaluation) and Bedrock agent orchestration (query planning, Athena execution, response generation), showing exactly where failures occur and automatically retrying transient errors. The CDK-based deployment creates consistent, repeatable infrastructure with automatic rollback on failures, enabling safe updates across development and production environments.

### Why are you using these services to support operational excellence?

We selected these services to minimize operational overhead while maximizing visibility into system behavior across diverse workloads (analytics, ML, AI agents). CloudWatch Logs eliminates the need for custom logging infrastructure and provides built-in log retention, search, and alerting capabilities that work consistently whether monitoring customer analytics queries or ML model training. X-Ray's automatic instrumentation captures distributed traces without requiring code changes, making it immediately useful for debugging the multi-step ML inference pipeline where a single prediction request touches 5+ services. The Glue Data Catalog's automatic schema discovery and versioning prevents the common operational problem of data pipelines breaking when source data formats change, which is critical when ingesting telemetry data from multiple vehicle types or customer data from various systems. Athena's query history serves dual purposes: performance optimization (identifying slow queries that need view optimization) and security auditing (tracking who accessed sensitive customer data). Step Functions provides visual debugging of complex workflows, reducing mean time to resolution from hours to minutes when ML training fails or Bedrock agents produce unexpected results. The infrastructure-as-code approach via CDK enables version-controlled infrastructure changes, peer review of modifications, and automated testing before production deployment. Together, these services create a self-documenting, self-monitoring platform that surfaces issues proactively (through CloudWatch alarms) rather than requiring manual investigation, while maintaining consistent operational practices across customer analytics, predictive maintenance, and AI agent workloads.

## Security

### What service(s) are you using to enhance security?

This platform uses AWS IAM, AWS Lake Formation, Amazon S3 encryption, AWS Secrets Manager, Amazon VPC, AWS KMS, and Amazon Bedrock Guardrails for security. IAM provides fine-grained access control with least-privilege policies for all services. Lake Formation enforces row-level and column-level security on customer data in the Glue Data Catalog, ensuring users only access data they're authorized to see. S3 buckets use server-side encryption with KMS customer-managed keys for data at rest. Secrets Manager stores database credentials for Aurora PostgreSQL (used by the Bedrock Knowledge Base) with automatic rotation. VPC isolates the Aurora cluster and SageMaker endpoints from public internet access. Bedrock Guardrails filter sensitive information in AI agent responses, preventing accidental exposure of PII.

### How do these services help the user with security?

These services implement defense-in-depth security across data storage, processing, and access layers. Lake Formation's fine-grained access control means a customer service representative querying the Customer 360 dashboard sees only their assigned region's data, while executives see aggregated data across all regions, all enforced at the data catalog level rather than requiring application-level filtering. S3 encryption with KMS ensures that even if bucket policies are misconfigured, data remains encrypted at rest and requires both S3 and KMS permissions to decrypt. Secrets Manager eliminates hardcoded database credentials in Lambda functions and Glue jobs, automatically rotating passwords without requiring code changes or downtime. VPC isolation prevents direct internet access to the Aurora database and SageMaker inference endpoints, requiring all access to flow through authenticated API Gateway endpoints. Bedrock Guardrails act as a safety layer for the AI agent, automatically redacting credit card numbers, social security numbers, and other PII patterns before responses reach users, even if the underlying data contains sensitive information.

### Why are you using these services to support security?

We selected these services to implement security controls that scale automatically and require minimal ongoing management. Lake Formation provides centralized data governance that applies consistently whether data is accessed through Athena, Amazon Q in Quick Suite, or Glue ETL jobs, eliminating the need to implement access controls separately in each service. KMS-managed encryption keys enable centralized key rotation and audit logging of all encryption/decryption operations through CloudTrail, supporting compliance requirements without custom key management infrastructure. Secrets Manager's automatic rotation prevents the common security risk of long-lived credentials while eliminating the operational burden of manual password updates across multiple services. VPC isolation follows the principle of defense-in-depth, ensuring that even if application-layer authentication fails, network-layer controls prevent unauthorized database access. Bedrock Guardrails provide a critical safety mechanism for generative AI workloads, where traditional rule-based filtering is insufficient because LLM responses are non-deterministic. Together, these services create multiple security layers that protect data throughout its lifecycle (at rest, in transit, in processing) while maintaining usability for authorized users and minimizing the risk of misconfigurations that could lead to data exposure.

## Reliability

### What service(s) are you using to enhance reliability?

This platform uses Amazon S3, AWS Glue, Amazon Athena, Amazon Aurora with Multi-AZ, Amazon SageMaker with multi-instance endpoints, AWS Step Functions, and AWS Lambda for reliability. S3 provides 99.999999999% durability for all data lake storage. Glue Data Catalog is a fully managed, highly available metadata store with automatic backups. Athena is serverless with no infrastructure to manage and automatic scaling for concurrent queries. Aurora PostgreSQL runs in Multi-AZ configuration with automatic failover for the Bedrock Knowledge Base vector store. SageMaker inference endpoints support multiple instances across availability zones with automatic traffic distribution. Step Functions provides automatic retry logic with exponential backoff for transient failures in ML training and Bedrock agent workflows. Lambda functions automatically scale to handle variable query loads and API requests.

### How do these services help the user with reliability?

These services eliminate single points of failure and automatically recover from infrastructure issues. S3's cross-AZ replication means customer data and telemetry data remain accessible even if an entire availability zone fails. Aurora's Multi-AZ deployment automatically fails over to a standby replica within 30 seconds if the primary database instance fails, ensuring the Bedrock Knowledge Base remains available for AI agent queries without manual intervention. SageMaker's multi-instance endpoints distribute prediction requests across multiple instances, so if one instance fails during tire failure prediction, traffic automatically routes to healthy instances without dropped requests. Step Functions' automatic retry logic handles transient failures in the ML training pipeline (such as temporary SageMaker capacity constraints or Glue job throttling) by retrying failed steps up to 3 times with exponential backoff, preventing entire workflows from failing due to temporary issues. Lambda's automatic scaling handles traffic spikes during peak hours (such as morning commute times when telemetry data ingestion peaks) without capacity planning or manual intervention.

### Why are you using these services to support reliability?

We selected these services to achieve high availability without the operational complexity of managing redundant infrastructure. S3's built-in durability and availability eliminate the need for custom replication logic or backup systems for the data lake. Aurora's managed Multi-AZ deployment provides automatic failover that would require significant engineering effort to implement with self-managed databases, while maintaining strong consistency for vector similarity searches in the Knowledge Base. SageMaker's managed endpoint scaling and health checks remove the burden of monitoring instance health and implementing traffic routing logic for ML inference. Step Functions' declarative retry policies make workflows resilient to transient failures without requiring custom error handling code in every Lambda function. The serverless nature of Athena, Lambda, and Glue means there are no servers to patch, no capacity to provision, and no infrastructure failures to recover from manually. Together, these services provide enterprise-grade reliability with minimal operational overhead, allowing teams to focus on business logic rather than infrastructure management while maintaining high availability for both customer-facing analytics and operational ML workloads.

## Performance Efficiency

### What service(s) are you using to enhance performance efficiency?

This platform uses Amazon Athena with partitioned tables, AWS Glue with Apache Spark, Amazon Aurora with pgvector indexing, Amazon SageMaker with GPU instances, Amazon S3 Intelligent-Tiering, and Amazon CloudFront (for Amazon Q in Quick Suite dashboards) for performance efficiency. Athena queries use partition pruning on date columns to scan only relevant data subsets. Glue ETL jobs leverage Spark's distributed processing to transform large telemetry datasets in parallel. Aurora's pgvector extension uses HNSW indexing for fast similarity searches across customer interaction embeddings. SageMaker training jobs use GPU instances (ml.g4dn family) to accelerate Random Cut Forest model training on large telemetry datasets. S3 Intelligent-Tiering automatically moves infrequently accessed historical data to lower-cost, lower-latency storage tiers. CloudFront caches QuickSight dashboard assets at edge locations for faster load times.

### How do these services help the user with performance efficiency?

These services optimize performance across data storage, processing, and retrieval patterns. Athena's partition pruning reduces query execution time from minutes to seconds by scanning only relevant date partitions when analyzing customer health trends over specific time periods, rather than scanning the entire dataset. Glue's Spark-based processing distributes telemetry data transformation across multiple workers, processing millions of vehicle sensor readings in parallel rather than sequentially. Aurora's pgvector HNSW indexing enables sub-second similarity searches across hundreds of thousands of customer interaction embeddings, making the Bedrock agent's retrieval-augmented generation fast enough for interactive conversations. SageMaker's GPU acceleration reduces ML training time from hours to minutes for the tire prediction model, enabling faster iteration on model improvements. S3 Intelligent-Tiering automatically optimizes storage costs without sacrificing performance, moving historical data to archive tiers while keeping recent data in high-performance storage. CloudFront's edge caching reduces Amazon Q in Quick Suite dashboard load times from seconds to milliseconds for geographically distributed users.

### Why are you using these services to support performance efficiency?

We selected these services to deliver optimal performance without over-provisioning resources or requiring manual performance tuning. Athena's serverless architecture with automatic partition pruning provides fast query performance without requiring users to manage cluster sizing or indexing strategies. Glue's managed Spark environment eliminates the complexity of configuring and tuning Spark clusters while automatically scaling workers based on job requirements. Aurora's pgvector extension provides production-grade vector search performance without requiring a separate vector database, reducing architectural complexity while maintaining low-latency retrieval for the Bedrock Knowledge Base. SageMaker's managed training infrastructure with GPU support delivers high-performance ML training without requiring expertise in GPU programming or cluster management. S3 Intelligent-Tiering's automatic optimization eliminates the need for lifecycle policies or manual data movement while ensuring frequently accessed data remains in high-performance tiers. CloudFront's global edge network provides consistent low-latency access to Amazon Q in Quick Suite dashboards regardless of user location, without requiring regional dashboard deployments. Together, these services deliver performance that scales with workload demands while minimizing the operational burden of performance monitoring, capacity planning, and infrastructure tuning.

## Cost Optimization

### What service(s) are you using to enhance cost optimization?

This platform uses serverless services (AWS Lambda, Amazon Athena, AWS Glue), Amazon S3 Intelligent-Tiering, Amazon Aurora Serverless v2, Amazon SageMaker with automatic scaling, AWS Step Functions, and Amazon Q in Quick Suite's pay-per-session pricing for cost optimization. Lambda charges only for actual compute time during query execution and data processing. Athena charges per query based on data scanned, incentivizing efficient query design with partitioning and columnar formats. Glue charges per second for ETL job execution with automatic worker scaling. S3 Intelligent-Tiering automatically moves data to lower-cost storage classes based on access patterns. Aurora Serverless v2 scales database capacity up and down based on actual workload, eliminating idle capacity costs. SageMaker inference endpoints use automatic scaling to match instance count with prediction request volume. Amazon Q in Quick Suite's pay-per-session pricing for readers means users only pay when actively viewing dashboards.

### How do these services help the user with cost optimization?

These services align costs directly with actual usage, eliminating waste from over-provisioned infrastructure. Lambda's per-invocation pricing means the platform only incurs compute costs when processing customer queries or telemetry data, not during idle periods overnight or weekends. Athena's per-query pricing combined with partitioned tables means analyzing a single day of customer interactions costs a fraction of scanning the entire dataset, incentivizing efficient query patterns. Glue's per-second billing with automatic scaling means ETL jobs use exactly the resources needed for each data volume, scaling down for small incremental updates and scaling up for large historical loads. S3 Intelligent-Tiering automatically reduces storage costs by 68% for historical data that's accessed infrequently (older than 90 days) without requiring manual lifecycle management or risking performance degradation. Aurora Serverless v2 scales down to minimal capacity during low-traffic periods (nights, weekends) and scales up during peak hours, eliminating the cost of maintaining full capacity 24/7. SageMaker's automatic scaling reduces inference costs by running fewer instances during low-demand periods while maintaining performance during peak prediction request volumes. Amazon Q in Quick Suite's pay-per-session model means occasional dashboard viewers (executives checking monthly KPIs) cost significantly less than full-time users.

### Why are you using these services to support cost optimization?

We selected these services to minimize costs without sacrificing performance or requiring constant cost management. The serverless architecture (Lambda, Athena, Glue) eliminates the largest source of cloud waste: idle infrastructure running 24/7 to handle peak loads that occur only a few hours per day. S3 Intelligent-Tiering's automatic optimization removes the operational burden of analyzing access patterns and creating lifecycle policies while ensuring optimal storage costs. Aurora Serverless v2's automatic scaling provides the cost benefits of right-sizing without requiring capacity planning or manual adjustments as workloads change. SageMaker's automatic scaling prevents the common anti-pattern of running large inference endpoints continuously to handle occasional prediction spikes. Amazon Q in Quick Suite's flexible pricing model allows cost-effective access for different user types (pay-per-session for occasional viewers, monthly subscriptions for power users) without forcing a one-size-fits-all pricing tier. The combination of per-use pricing, automatic scaling, and intelligent tiering creates a cost structure that naturally optimizes as usage patterns change, without requiring dedicated cost optimization engineering or constant monitoring of resource utilization.

## Sustainability

### What service(s) are you using to enhance sustainability?

This platform uses serverless services (AWS Lambda, Amazon Athena, AWS Glue), Amazon S3 Intelligent-Tiering, Amazon Aurora Serverless v2, AWS Graviton-based instances, and AWS Regions with renewable energy commitments for sustainability. Serverless services eliminate idle compute resources by running only when processing requests. S3 Intelligent-Tiering reduces storage energy consumption by moving infrequently accessed data to more energy-efficient storage classes. Aurora Serverless v2 minimizes database infrastructure by scaling capacity to match actual workload. Lambda functions and Glue jobs can run on Graviton processors, which provide up to 60% better energy efficiency than comparable x86 processors. The platform can be deployed in AWS Regions powered by renewable energy (such as us-west-2 Oregon, which runs on 95% renewable energy).

### How do these services help the user with sustainability?

These services minimize environmental impact by reducing energy consumption and maximizing resource utilization. Serverless Lambda and Glue eliminate the carbon footprint of idle servers running 24/7, ensuring compute resources are only active when processing customer queries or telemetry data. S3 Intelligent-Tiering's automatic data movement reduces storage energy consumption by storing historical data (older than 90 days) in more energy-efficient storage classes that use less power per GB. Aurora Serverless v2's automatic scaling reduces database infrastructure energy consumption by scaling down to minimal capacity during low-traffic periods rather than maintaining full capacity continuously. Graviton-based Lambda functions and Glue workers deliver the same performance with 60% less energy consumption compared to x86 instances, directly reducing the carbon footprint of data processing workloads. Deploying in Regions with renewable energy commitments ensures the platform's energy consumption is increasingly powered by wind, solar, and hydroelectric sources rather than fossil fuels.

### Why are you using these services to support sustainability?

We selected these services to minimize environmental impact without compromising performance or increasing operational complexity. The serverless architecture inherently maximizes resource utilization by sharing infrastructure across multiple customers and eliminating idle capacity, which is the most significant source of wasted energy in traditional data centers. S3 Intelligent-Tiering's automatic optimization reduces storage energy consumption without requiring manual data management or risking performance degradation from overly aggressive archival policies. Aurora Serverless v2's automatic scaling ensures database infrastructure matches actual workload demands, eliminating the common practice of over-provisioning databases to handle peak loads that occur only occasionally. Graviton processors provide better energy efficiency without requiring code changes or performance trade-offs, making sustainability improvements accessible without additional engineering effort. The ability to deploy in Regions with renewable energy commitments allows organizations to reduce their carbon footprint by choosing locations where AWS has invested in renewable energy infrastructure. Together, these services enable organizations to reduce their environmental impact while maintaining the performance and reliability required for production workloads, demonstrating that sustainability and business objectives can align rather than conflict.
